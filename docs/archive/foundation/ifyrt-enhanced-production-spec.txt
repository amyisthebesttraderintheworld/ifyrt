IFYRT / TriFi — Production Design Specification (Enhanced)
0. Purpose
This document defines a production-ready, multi-tenant trading simulation and execution platform with strict separation between simulation and live trading. It is designed to be immediately consumable by an engineering agent (e.g., Codex) and includes:
• Concrete service boundaries n- Deterministic simulation guarantees
• Realistic execution modeling
• Security-first live trading design
• Event-driven orchestration
This version restores critical design intent, reasoning, and system philosophy that were previously stripped out.
1. Core Philosophy (Non-Negotiables)
1.1 Simulation is the Product
Simulation is not a feature — it is the primary product surface.
• Must be deterministic (same inputs → same outputs)
• Must be high-fidelity (real order book behavior)
• Must be behaviorally identical to live execution
1.2 Parity Principle
If a strategy works in simulation, it should behave the same in live — within market variance.
Achieved via:
• Shared execution interface
• Identical order semantics
• Same fee + slippage modeling
1.3 Hard Separation: Simulation vs Live
LayerSimulationLiveExecutionSimulated engineExchange APIsFundsVirtualRealKeysNoneRequiredRiskZeroReal 
No shared state. No crossover. Ever.
2. System Architecture
2.1 High-Level Topology
Telegram Bot ↓ API Gateway (Stateless) ↓ Event Bus (n8n + Realtime) ↓ Core Services: - Simulation Engine - Market Data Ingestor - Strategy Runtime - Live Execution Engine - Copy Trading Layer ↓ Persistence (Supabase) ↓ Optional Dashboard (Vercel) 
2.2 Architectural Style
• Event-driven
• Stateless services
• Horizontal scaling
• Worker isolation per simulation
3. Telegram Control Plane
3.1 Role
Telegram is the authoritative control interface.
Why:
• Zero-friction UX
• No additional auth surface
• Forces backend consistency
3.2 Responsibilities
• Parse commands
• Emit structured events
• Never execute logic
3.3 Event Contract
All commands emit a strict event schema:
interface IfyrtEvent { event_id: string; event_type: string; timestamp: string; user: { telegram_id: number }; payload: Record<string, any>; } 
4. Event Orchestration Layer (n8n)
4.1 Role
Acts as the central nervous system.
Responsibilities:
• Route events
• Enforce subscription
• Trigger workers
• Handle retries
4.2 Critical Insight
No service calls another service directly.
All communication flows through:
• Supabase (state)
• n8n (control)
5. Market Data Layer
5.1 Requirements
• Real-time ingestion (WebSocket)
• Historical OHLCV storage
• Order book reconstruction
5.2 Design Choice
Order book stored as JSONB initially for speed of development.
Tradeoff:
• ✔ Fast iteration
• ✖ Not optimal at scale (future: TimescaleDB)
6. Simulation Engine (Core)
6.1 Execution Model
Market Data → Strategy → Signal → Matching Engine → Fill → Metrics 
6.2 Determinism Rules
• No randomness
• No async race conditions
• Fixed latency modeling
6.3 Matching Engine Guarantees
Market Orders
• Walk real order book
• Slippage emerges naturally
Limit Orders
• Fill only when price crosses
Fees
• Applied per exchange rules
7. Strategy Runtime
7.1 Contract
on_tick(candles, position) → signal 
7.2 Constraints
• Pure function only
• No I/O
• No randomness
Why: Ensures reproducibility and sandbox safety.
8. Unified Execution Interface
8.1 Core Idea
Same function drives both worlds:
execute(order) 
8.2 Adapters
ModeAdapterSimulationMatching EngineLiveExchange API 
9. Live Trading Engine
9.1 Isolation
Runs as a completely separate service.
9.2 Requirements
• JWT-authenticated
• Paid-tier gated
• Circuit breaker protected
10. Secure Key Management
10.1 Design
• Keys stored in Vault (encrypted)
• Only decrypted in-memory
• Never logged
10.2 Critical Rule
If a key touches disk or logs, the system is compromised.
11. Circuit Breakers (Risk System)
11.1 Triggers
• Max drawdown exceeded
• Abnormal execution behavior
11.2 Action
• Immediate halt of live trading
• Session marked circuit_broken
12. Copy Trading (Simulation Only by Default)
12.1 Model
• Leaders emit signals
• Followers simulate independently
12.2 Security
• No strategy code sharing
• Signals only
