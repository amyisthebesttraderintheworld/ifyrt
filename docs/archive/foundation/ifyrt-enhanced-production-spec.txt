# Ifyrt / TriFi — Production Design Specification

## 0. Purpose

This document defines a production-ready, multi-tenant trading simulation and execution platform with strict separation between simulation and live trading. It covers:
- Concrete service boundaries
- Deterministic simulation guarantees
- Realistic execution modeling
- Security-first live trading design
- Event-driven orchestration

## 1. Core Philosophy

### 1.1 Simulation is the Product
Simulation is the primary product surface.
- Must be deterministic (same inputs → same outputs)
- Must be high-fidelity (real order book behavior)
- Must be behaviorally identical to live execution

### 1.2 Parity Principle
Strategies should behave the same in live trading as they do in simulation, within market variance.
- Shared execution interface
- Identical order semantics
- Consistent fee and slippage modeling

### 1.3 Strict Separation
There is no shared state or crossover between simulation and live execution.

| Layer | Simulation | Live |
| :--- | :--- | :--- |
| Execution | Simulated engine | Exchange APIs |
| Funds | Virtual | Real |
| Keys | None | Required |
| Risk | Zero | Real |

## 2. System Architecture

### 2.1 High-Level Topology
Control flows from the Telegram Bot through a stateless API Gateway, then via n8n and Realtime event buses to core services:
- Simulation Engine
- Market Data Ingestor
- Strategy Runtime
- Live Execution Engine
- Copy Trading Layer
- Persistence (Supabase)
- Optional Dashboard (Vercel)

### 2.2 Architectural Style
- Event-driven and stateless services
- Horizontal scaling
- Worker isolation per simulation

## 3. Telegram Control Plane

### 3.1 Role
Telegram is the authoritative control interface.
- Minimal friction UX
- Integrated authentication
- Centralized backend consistency

### 3.2 Responsibilities
- Parse commands
- Emit structured events
- Forward to orchestration layer without executing logic

### 3.3 Event Contract
```ts
interface IfyrtEvent {
  event_id: string;
  event_type: string;
  timestamp: string;
  user: {
    telegram_id: number;
  };
  payload: Record<string, any>;
}
```

## 4. Event Orchestration Layer (n8n)

### 4.1 Role
Acts as the central nervous system of the platform.
- Route events
- Enforce subscription gates
- Trigger workers
- Handle retries and failures

### 4.2 Communication
Services do not call each other directly; all communication flows through Supabase (state) and n8n (control).

## 5. Market Data Layer

### 5.1 Requirements
- Real-time ingestion via WebSocket
- Historical OHLCV storage
- Order book reconstruction

### 5.2 Storage
Order book data is stored as JSONB for rapid development, with plans to migrate to a time-series optimized store at scale.

## 6. Simulation Engine

### 6.1 Execution Model
`Market Data → Strategy → Signal → Matching Engine → Fill → Metrics`

### 6.2 Determinism Rules
- No randomness
- No async race conditions
- Fixed latency modeling

### 6.3 Matching Engine
- **Market Orders:** Walk the real order book; slippage emerges naturally from depth.
- **Limit Orders:** Fill only when price crosses.
- **Fees:** Applied per exchange rules.

## 7. Strategy Runtime

### 7.1 Contract
`on_tick(candles, position) → signal`

### 7.2 Constraints
- Pure functions only
- No I/O or external calls
- No randomness
- Ensures reproducibility and sandbox safety.

## 8. Unified Execution Interface

Both simulation and live execution use the same `execute(order)` interface, driven by different adapters:
- **Simulation Adapter:** Matching Engine
- **Live Adapter:** Exchange API

## 9. Live Trading Engine

- Runs as a completely isolated service.
- Requires JWT authentication and a paid-tier subscription.
- Protected by circuit breakers and risk limits.

## 10. Secure Key Management

- Keys are stored in an encrypted vault.
- Decryption occurs only in-memory within the execution environment.
- Keys are never logged or persisted to disk.

## 11. Circuit Breakers

- **Triggers:** Max drawdown exceeded or abnormal execution behavior detected.
- **Action:** Immediate halt of live trading; session marked as `circuit_broken`.

## 12. Copy Trading

- Leaders emit signals; followers simulate independently.
- No strategy code is shared; only signals are routed.
- Default mode is simulation only.

## 13. Subscription Model

| Tier | Access |
| :--- | :--- |
| Free | Backtesting only |
| Trial | Simulation + Live Toggle |
| Paid | Full system access |

## 14. Persistence Layer

- **Core Tables:** `users`, `subscriptions`, `simulations`, `trades`, `api_keys`, `market_data`.
- **Constraint:** `trades.is_live` must always be filtered to prevent mixing live and simulated records.

## 15. Observability

- Centralized logging, metrics, and alerting.
- Key metrics: Simulation latency, fill accuracy, and drawdown rates.
